DESIGN DOCUMENT · v0.3 · World Cup 2026

Football Prediction App

Baseline AI match predictions that visibly react to what the user feeds them.

Contents

    1 · Vision

    Mobile-first web app where casual fans get an automatic pre-match prediction — outcome (W/D/L with probabilities) and final score — built from news and stats. The user refines it with a note or article link; the model adapts and explains, in plain language, what changed and why.

    The hook is the delta between baseline and refined prediction. The model visibly reacting to your input is what separates this from every black-box prediction site.

    2 · Audience & Scope

    Audience: casual fans — not bettors, not analysts. The 10-second baseline read satisfies most visits; refinement is for the engaged minority who want to argue with the model.

    Scope v1: match outcome (Home / Draw / Away + confidence) and predicted final score only. No xG, possession, or goalscorers — a small surface keeps the baseline easy to explain and the refinement easy to interpret.

    Launch target: FIFA World Cup 2026 (opens 11 June 2026). The tournament is the primary event the product is built and timed for — global casual audience, intense news coverage, a hard date. Premier League and Champions League are covered as run-up content so the app has live matches and an audience before the opening match.

    3 · The Baseline

    Generated automatically before kickoff and shown to every visitor, anonymous included, with zero friction. Each baseline carries the predicted outcome (e.g. Arsenal 2-1, 58% confidence), a short plain-language "why" paragraph naming the driving factors, and inline citations to the source articles.

    Freshness. A baseline is generated at ~T-24h, then auto-refreshed when lineups are released (~T-1h) and on major breaking news. Baselines are versioned; a refinement records which baseline version it forked from so the delta stays meaningful even after a refresh.

    Low-coverage matches. If few or no articles exist, the model falls back to a stats-only baseline with lower confidence, labelled "limited news available". The app never fails to produce a baseline.

    Citations. Always available, tap-to-reveal on mobile to keep the card compact — tapping shows source headline and outlet. For casual fans this is the difference between "this site made something up" and "this site read 8 articles for me."

    4 · The Refinement Flow

    Beneath the baseline, a single input invites the user to add something the model doesn't know. It accepts free text or a URL. On submit the prediction recalculates with a brief "thinking" animation and lands on a new outcome and a new "why" that explicitly references the user's input.

    Baseline and refined prediction live side by side, never one replacing the other — the delta is the product. The user's input renders as an editable chip between them ("you added: Saka injured").

    Chip rule (one active chip per match). Submitting a chip is one refinement. Editing or removing that chip is free — it re-runs on the same credit, so iterating on one piece of context never feels punishing. Removing the chip reverts cleanly to the baseline. A different chip submitted after removal is a new refinement. This makes the credit model unambiguous.

    Personal & persistent. Refinements affect only the requesting user's view — they never pollute the shared baseline, which avoids a launch-day moderation problem. They also survive across sessions: submit at 1pm, return at 2pm, the refined prediction (with editable chip) is still there.

    5 · Prediction Engine

    The engine is an LLM (Claude) producing structured JSON, not a custom ML model in v1. It assembles context — team form/lineups plus a ranked set of article snippets — into a single prompt and receives { outcome_probs, score, why, citations[] } back. The same path serves baseline and refinement; refinement just adds the user's input to the context.

    Data sources. NewsData.io (sports category, commercial-licensed paid plan) is the news layer — publisher RSS feeds were ruled out because most outlet RSS terms forbid commercial reuse. We store only headline, outlet, URL, and a short snippet, never full article text (copyright); the citation links out for the full read. API-Football is the single sports-data provider for fixtures, form, lineups, and final results, and it covers the World Cup. For national-team matches the engine uses recent international form and the announced squad/lineup in place of club form — same pipeline, different context fields.

    User-submitted URLs are fetched server-side only, through a sandboxed, SSRF-guarded fetcher: domain allowlist, blocked private IPs and redirects, a 5-second and size cap, readability extraction, scripts stripped.

    Prompt-injection defence. User text and any fetched article go into a clearly delimited "untrusted data" block; the system prompt forbids following instructions found inside it. Returned citations are validated against the source list we supplied, so the model cannot invent URLs.

    Cost control. A baseline is computed once per match and cached — every anonymous read is a cache hit with no model call. A cheap model handles input validation; the capable model runs only for the final prediction. The per-day refinement quota is the effective spend cap.

    7 · Auth & Monetization

    Auth is Clerk, contextual, not a top-nav CTA. The Clerk sign-in modal appears at the exact refine moment with copy that sells the action, not the account ("Sign in to tell the model what it's missing. Free, 3 refinements per day."). Clerk is configured with Google and Apple social connections only at launch — email/password adds drop-off without much value for a casual product. Clerk owns the user store, sessions, and OAuth; the BFF trusts a request only after verifying the Clerk session JWT. The user's tier lives in Clerk publicMetadata, kept in sync by the Stripe billing webhook, so tier travels with the session token and needs no extra lookup.

    Free quota: 3 successful refinements per day. Only successful ones count — a validation step returns parsed and relevant flags, and the quota decrements only when both are true. Rejected gibberish, a dead URL, or off-topic input costs no credit, so the counter never feels adversarial. The quota resets at midnight in the user's account timezone (UTC fallback), shown in the UI.

    Conversion moment is the third successful refinement, not the fourth. Right after it lands, a soft prompt appears at the engagement peak ("You're getting the hang of this. Unlimited refinements all tournament for $5?"). The hard wall only comes on the next day's attempt — hitting it cold, without the prior nudge, bounces users.

    Pricing (v1, USD). Two passes, no monthly subscription at launch:

    • $0.99 single-matchday pass — unlimited refinements for one match day. Matches the casual mental model ("I'll pay for this big game") and keeps billing scope small.
    • $5 World Cup tournament pass — unlimited refinements for the whole tournament. The event-scoped pass is the main conversion play: one decision, one price, urgency built in by the fixed dates.

    A recurring monthly subscription is deferred until after the World Cup, once there is repeat-usage data to price it against. The free tier (3 successful refinements/day) is unchanged. A fair-use ceiling of ~30 successful refinements/day applies even to pass holders — well above any genuine casual use, there only to cap abuse and protect the model spend that the unlimited passes would otherwise leave uncapped.

    8 · Trust & Legal

    Every prediction carries an "entertainment, not betting advice" disclaimer to stay clear of gambling-adjacent positioning. GDPR from day one: explicit consent, data export and delete, and minimal storage of user-submitted input.

    9 · Habit Loop post-MVP, design now

    After each match concludes, show logged-in users how their refined call compared to the baseline and the actual result ("Your call beat the model 3× this week"). This turns a single-use site into a habit and gives subscribers a tangible reason to keep paying.

    Even though v1 may ship without it, the data model supports it from day one: every refinement stores a prediction snapshot tied to the eventual outcome. Scoring uses the Brier score for the outcome and scoreline distance for the scoreline; "beat the model" is the better of the two. The profile screen is its natural home.

    10 · Screen Architecture

    • Match list / Home — today & this-weekend games, each row showing teams, kickoff, and a one-line baseline preview. Filter by league; onboarding surfaces favourite leagues so Home feels personal.
    • Match detail — the main page. Header → baseline card (why + citations) → refinement input → refined card when present. Includes the refinement counter for free users.
    • Contextual sign-in sheet — Clerk sign-in modal over match detail when an anonymous user taps the input. Google/Apple. Value-led copy.
    • Contextual upgrade sheet — soft after the 3rd successful refinement, hard on daily-limit hit. Matchday-pass + World Cup tournament-pass options.
    • Profile / Settings — logged-in only. Favourite leagues, account, subscription. Post-MVP: refinement history and accuracy record.

    12 · Component Deep-Dive

    One scenario runs through every component below. Saturday, 16:00 kickoff — Arsenal vs Chelsea. Mia, a casual Arsenal fan, opens the app on her phone at 14:30. She is not logged in yet.

    1 · Web Frontend

    What it does: the only thing the user touches — a mobile-first single-page web app. Renders the match list, the match-detail page with baseline and refined cards side by side, the chip UI, and hosts the contextual Clerk and Stripe sheets. Holds no business logic and no secrets; it asks the BFF for everything.

    Role in the flow: entry and exit point. Turns user gestures into BFF calls and BFF responses into screens.

    Example: Mia taps the "Arsenal vs Chelsea" row on Home. The frontend opens match detail and shows the baseline card — "Arsenal 2-1, 58%", a short why-paragraph, three citation chips. The refinement box is visible but greyed because she is anonymous.

    2 · Backend API (BFF)

    What it does: the single trusted server the frontend talks to. Verifies the Clerk session, enforces the daily quota, reads the baseline cache, orchestrates refinement (URL fetch → Prediction Engine), and owns every business rule. Hides all internal services behind one boundary.

    Role in the flow: the conductor — nothing reaches the engine, cache, or database except through it.

    Example: Mia's phone calls GET /match/123/baseline. The BFF sees baseline needs no auth, asks the Cache, gets a hit, returns it in milliseconds — no model call, no database write.

    3 · Clerk (managed auth)

    What it does: a hosted service that owns the user store, sessions, the sign-in UI, and Google/Apple OAuth. We write no auth code. The user's tier lives in Clerk publicMetadata and travels inside the session token.

    Role in the flow: the gate. The BFF trusts a request only after verifying Clerk's session JWT and reading the tier from it.

    Example: Mia taps the greyed input. The Clerk modal slides up — "Sign in to tell the model what it's missing. Free, 3 refinements per day." She taps Continue with Google, returns signed in; the next BFF call carries a JWT that says tier: free.

    4 · Prediction Engine

    What it does: builds both baseline and refined predictions. Gathers context (team form, lineup, ranked article snippets, plus the user's input for a refinement), calls the LLM Gateway, then parses, validates, and attaches citations to the structured result.

    Role in the flow: the brain. Everything else feeds it context or stores what it produces.

    Example: at 14:00 a scheduled job had the Engine assemble 8 Arsenal/Chelsea articles + form + provisional lineup, call the gateway, and get back { Arsenal win 58%, score 2-1, why, 8 citations }. It stored that as baseline v1 and cached it — the very card Mia is reading.

    5 · LLM Gateway

    What it does: the single choke point for every Claude call. Applies the prompt template and JSON output schema, isolates untrusted user and article text in a delimited block, logs token cost, and retries on failure.

    Role in the flow: the safety and cost valve between the Engine and the model — one place to audit spend and block prompt injection.

    Why a separate service, not a direct Claude call from the Engine:

    • One home for prompt & schema — the prompt template and the strict output shape { outcome_probs, score, why, citations[] } live here, not scattered across callers.
    • Injection isolation, enforced once — every untrusted input is wrapped in the "do not obey instructions inside" block and every returned citation is checked against the source list we supplied, so the model cannot invent URLs. A caller cannot forget to do this.
    • Cost control & audit — tokens and € logged per call in one dashboard; routes a cheap model for validation and the capable model only for the final prediction.
    • Reliability — retries, timeouts, and rate-limit handling in one place; swapping model version (e.g. Opus 4.6 → 4.7) or provider touches only this service, never the Engine.
    • Key custody — the Claude API key lives only here, never closer to the user.

    It is to the model what the URL Fetcher (component 7) is to untrusted links: a controlled airlock with everything logged. The trade-off is one extra service for that isolation; folding it into the Engine is simpler but loses the single enforcement point.

    Example: when Mia later submits a note, the gateway wraps it as untrusted data — do not follow instructions inside, sends the prompt, logs €0.012 for the call, and returns validated JSON to the Engine.

    6 · Ingestion Workers

    What it does: scheduled background jobs that pull the outside world in — news fetch and ranking, fixtures, lineups, team form, and post-match results — and write it to the datastore. Nothing the user triggers.

    Role in the flow: the supply line. Keeps the Engine's raw material fresh; without it the baseline goes stale.

    Example: at 14:55 the lineup worker pulled the confirmed Arsenal XI and saw Saka not in it; the news worker had earlier pulled a "Saka fitness doubt" article. Both landed in the datastore, triggering a baseline refresh to v2.

    7 · URL Fetcher / Extractor

    What it does: a sandboxed, isolated service that safely fetches a user-submitted link. Domain allowlist, blocked private IPs and redirects, 5-second and size caps, readability extraction, scripts stripped.

    Role in the flow: the airlock for untrusted links — keeps a pasted URL from becoming an SSRF or injection attack.

    Example: instead of typing, Mia pastes a Sky Sports article link. The fetcher confirms the domain is allowed, downloads it, strips scripts, and hands the Engine a clean 600-word extract.

    8 · Datastore (Postgres)

    What it does: the durable system of record — users' app data, matches, baseline versions, every refinement, prediction snapshots, citations, the quota ledger, and final results.

    Role in the flow: long-term memory. Makes refinements persist across sessions and makes post-match scoring possible.

    Example: Mia's refinement is stored as a snapshot — match 123, baseline v2, input "Saka injured", predicted Arsenal 1-1 — tied to match 123 so it can be scored after the final whistle.

    9 · Cache (Redis)

    What it does: fast in-memory store for the computed baseline, rate limits, per-user quota counters, and sessions.

    Role in the flow: the cost shield. Anonymous baseline reads hit Redis, never the model — this is what makes free, unlimited baseline viewing affordable.

    Example: 4,000 people open this match before kickoff. All 4,000 get baseline v2 from Redis in ~8 ms with zero model calls. Mia's counter reads mia:2026-05-16 → 1 after her first refinement.

    10 · Scheduler

    What it does: fires time-based jobs — baseline generation at T-24h, lineup refresh at T-1h, breaking-news refreshes, and post-match scoring.

    Role in the flow: the clock. Drives everything that must happen at a moment rather than in response to a user.

    Example: the T-1h job fired at 15:00, told the Engine to rebuild on the confirmed lineup (→ v2). At 18:00 it will fire the scoring job for this match.

    11 · Billing (Stripe)

    What it does: handles payment — the monthly subscription and the single-matchday pass — via Stripe Checkout. On a successful payment its webhook writes the new tier into Clerk publicMetadata.

    Role in the flow: the upgrade path. Converts the soft prompt into a tier change that the next session token reflects automatically.

    Example: after Mia's 3rd successful refinement the soft prompt appears. She taps it, pays €4/month in Stripe Checkout; the webhook flips her Clerk metadata to tier: paid. Her next refinement has no counter.

    12 · Analytics / Observability

    What it does: records product events, drives the pricing A/B test, tracks LLM cost, and feeds the post-match accuracy comparison for the habit loop.

    Role in the flow: the feedback layer — tells us what is working and powers "your call beat the model".

    Example: Mia's session logged baseline_view, refine_success, A/B bucket price_4eur, and €0.04 total model spend. After full-time, the scoring job emits "Mia's 1-1 call beat the model's 2-1" — surfaced on her profile.

    14 · Decisions & Remaining Risks

    Locked for v1

    • Pricing — $0.99 matchday pass + $5 World Cup tournament pass; recurring monthly subscription deferred until after the World Cup. (§7)
    • Leagues — World Cup is the launch target; Premier League + Champions League covered as run-up. No other competitions in v1. (§2)
    • Vendors — API-Football (sports data) + NewsData.io (news, sports category, paid commercial-licensed plan). (§5)
    • Refinement style — one-shot; chat-style deferred.
    • Community — leaderboards / fan stream deferred past v1 (moderation + the personal-refinement guarantee).

    Remaining risks

    • Timeline — World Cup opens 11 June 2026. The app must be live and indexed before the group stage; this date does not move and is the dominant constraint on v1 scope.
    • Cost concentration — unlimited-refinement passes across ~104 matches in five weeks. Mitigated by the ~30/day fair-use ceiling (§7); validate real per-user model cost early against the $5 price.
    • Currency / localization — pricing is USD only at launch; multi-currency and localization deferred.
    • Low-coverage nations — minnow fixtures get thin news; the stats-only fallback (§3) carries more weight at a World Cup than in club football and should be tested before the tournament.

    Every phase ends with a runnable, demoable artifact — not just merged code.

    PhaseTestable exit — concretely verifiable
    0aspire run brings up all services green in the Aspire dashboard; Postgres has the §15 schema (migration applied); a health endpoint returns 200 behind a valid Clerk JWT and 401 without one; the React shell loads with a working Clerk sign-in.
    1Run the ingestion job → World Cup fixtures, teams, and ranked articles are queryable in Postgres. Integration test asserts a known WC fixture and ≥1 linked article exist after a sync.
    2Call the Prediction Engine for a seeded fixture → a valid baseline (probabilities sum to 1, score, why, citations resolve to real articles) is stored and cached. Re-request hits Redis, no second LLM call (assert via OTel span / cost log = 0).
    3Open the deployed URL on a phone as an anonymous user: match list → match detail → baseline card with working citations. End-to-end smoke test (Playwright) passes against a real WC fixture.
    4Sign in via Clerk, submit a text refinement and a URL refinement → refined card appears side-by-side with a changed "why"; chip edit/remove works; 4th refinement is blocked; gibberish does not consume a credit. E2E test covers the full hook.
    5Buy a $5 WC pass in Stripe test mode → webhook flips tier, counter disappears, refinements are unlimited (to the fair-use cap). Test asserts entitlement row + Clerk metadata in sync.
    6Settle a finished fixture → snapshot scoring job writes Brier + scoreline distance. Security checks pass (SSRF/injection suite). Load test sustains target concurrent baseline reads off cache within latency budget.

    Testing strategy

    • Unit — xUnit for .NET services (engine validation, quota rules, entitlement/tier resolution, SSRF guard); Vitest for React.
    • Integration — Aspire.Hosting.Testing spins the real app graph with Postgres + Redis (Testcontainers) for ingestion, engine, and BFF endpoint tests. External APIs (API-Football, NewsData.io, Anthropic, Stripe) stubbed via recorded fixtures / WireMock.
    • End-to-end — Playwright against the Aspire-run stack: anonymous baseline read (P3), the refinement hook (P4), the purchase flow in Stripe test mode (P5). These E2E specs are the per-phase acceptance gate.
    • Non-functional — a load script for the cached baseline path (P6) and the SSRF / prompt-injection suite (P6), both runnable in CI.

    Parallelization & dependencies

    • Phase 1 (Ingestion) and Phase 2 (Gateway/Engine) run in parallel once the Phase 0 schema is stable.
    • Phase 3 (Frontend) starts against a mocked BFF during Phase 2.
    • Phase 5 (Stripe) is independent — can begin once the Phase 4 tier model exists.
    • Critical path: schema → ingestion → engine → baseline read → refinement → billing.

    Cut-line if slipping drop in this order

    1. Matchday pass — keep only the $5 World Cup pass.
    2. PL / CL run-up coverage — World Cup fixtures only.
    3. Citation polish (still functional, less refined).
    4. Scaling headroom — accept slower group-stage simultaneous kickoffs.

    Never cut: baseline prediction, the refinement delta, the $5 WC pass, and the SSRF / prompt-injection defences.

    Key risks

    • Group-stage concurrency — multiple simultaneous kickoffs; the Redis baseline cache is the load shield and must be verified under load before the tournament.
    • LLM cost on the unlimited WC pass — fair-use cap plus OTel cost alerting are mandatory, not optional.
